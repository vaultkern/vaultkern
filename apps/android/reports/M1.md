# Android M1 — unlock

- 日期：2026-07-23（Asia/Shanghai）
- 分支：`codex/android-m1-unlock`（基于 `main@a514117`）
- 契约：`docs/design/000` r16、`002`、`003` r15（规则 7）、`007`
- 结论：M1 验收项通过；共享接口只增加 Android 所缺的窄入口，既有
  Windows/macOS 调用保持不变。

## 交付

- Android 14+ Compose resident app 直接持有一个进程内 `VaultSession`；UI、
  Biometric Activity 与后续 Android system service 共用该 session、runtime state
  和核心 writer lock。
- Android Keystore AES-GCM unlock blob 使用 auth-per-use `BIOMETRIC_STRONG`、
  `setInvalidatedByBiometricEnrollment(true)`、StrongBox 优先及 TEE 回退；密文、
  key alias 和实际 `KeyInfo.securityLevel` 作为一个 app-private `AtomicFile` 记录提交。
- 新 key 先准备，密文记录原子提交后才清理旧 key；启动调和清除未提交写入和孤儿
  alias。撤销删除密文与全部 `vaultkern.unlock.*` key；删除未完成会保留待重试状态。
- enrollment 不再由“记录/alias 存在”推断：每个当前 vault 都初始化 decrypt cipher
  并执行硬件策略探测。永久失效、缺 key、坏 tag 或 release 下非硬件 key 均删除并
  分类为 `invalidated`；取消和暂时性失败保留原项。安全级别也严格按当前 vault 查询。
- 交互解锁携带的 fresh credential 优先替换旧 blob，因此外部改密后不必先经历一次
  失败的 biometric unlock。若注册在 fresh UV 后失败或取消，尚未提交的 authenticated
  cipher/key 会立即销毁，旧的已提交 blob 保持不变。
- quick-unlock 开关只提交 desired state；创建/撤销在提交后的幂等调和及每次解锁后
  执行。提交失败时持久状态不变、用户草稿保留并标记未保存；提交成功但调和失败时
  明确显示 pending，下一调和点继续收敛。调和串行化，旧注册不会越过新撤销。
- 本地 vault 通过 `ACTION_OPEN_DOCUMENT` 和持久 read/write URI grant 选择。所选 URI
  始终是权威源；app-private 加密工作文件只用于现有核心的原子路径语义，并在每次锁定
  状态读取前从当前 URI 刷新。只刷新当前 vault，已撤权的无关历史 URI 不会阻塞它；
  工作文件丢失时可由权威源重建。M2 的保存发布仍回到同一 URI。
- key file 不再以 raw path 或 `/proc/self/fd` 进入核心。每次 unlock/enroll 从
  `ContentResolver` 新开一次流，以 `VaultKernSensitiveBytes` 把内容移交 Rust；
  Rust 解析并立即丢弃原始字节，只把 002 定义的 content-hash contribution 放入
  master credential。云 provider 的 pipe FD 也适用，且不写明文临时文件。
- UI 密码不进入 Compose `String`、`StateFlow` 或 DTO：使用禁用保存/Autofill 的原生
  clearable editor，点击交互解锁即覆盖并清空；切换本地 vault、改走 biometric、
  错误、取消和 ViewModel coroutine 未启动等路径也清理所有权。窗口使用 `FLAG_SECURE`。
- UniFFI Kotlin callback 在 lowering 完成或抛错后关闭返回的
  `VaultKernSensitiveString/Bytes` owner；所有中间 byte copy 均在 `finally` 清零，
  sensitive owner factory 即使复制失败也会清零输入。日志、状态和异常文案只记录类型，
  不承载 key material。
- 核心既有 unlock 路径继续负责文件 HMAC 有效性、换盐 cache miss 时在 resident app
  内重跑 KDF 并刷新 `{master credential, cached transformed key}`；Android session
  继续使用 mobile KDF policy（Argon2 `>128 MiB`、AES-KDF `>600M` 拒绝）。

## 小型加法绑定

每组缺失能力独立成小提交，未改变既有签名：

- `VaultSources.addLocalVault`：登记已有 private working path，而不为“选择”加载/保留
  加密 snapshot。
- `VaultSources.currentLocalVaultPath`：只读取得当前 local working path，供 Android
  定位唯一应刷新的 SAF authority。
- `VaultUnlock.unlockCurrentWithKeyFile` / `enrollWithKeyFile`：以 sensitive bytes
  传入同一份 key-file 语义，旧的 path-based API 保持原样。
- Kotlin callback sensitive-return owner 的生成后清理属于绑定卫生修复，不改变接口。

## 验证

设备：`VaultKern_M0_API34` AVD，Pixel 6 profile，Android 14 / API 34，
Google APIs x86_64。该 AVD 无 StrongBox，实际 Keystore security level 为 `SOFTWARE`；
debug 仅为模拟器验证允许 software，release 会拒绝并删除 software blob。因此本结果
没有外推为物理设备 TEE/StrongBox 证明。

M0 的真实系统 ceremony 证据按任务要求直接沿用、未重跑：同一 AVD 上
BiometricPrompt AES-GCM round-trip 通过，新增指纹后 key 永久失效且 blob 删除。
M1 产品代码另以真实 `AndroidKeyStore` 的 `KeyInfo` 仪器测试确认 auth-per-use、
biometric-enrollment invalidation 标志与记录的安全级别一致。

执行结果：

- `cargo test --locked -p vaultkern-runtime -- --test-threads=1`
  — 624 passed / 5 ignored（11 suites）；覆盖共享 runtime 原有行为，确认新增 bytes
  入口没有改变 path-based 调用。
- `cargo test --locked -p vaultkern-uniffi -- --test-threads=1`
  — 38 passed（4 suites）；其中 key-file protected fixture 完成
  unlock → enroll → lock → blob unlock round-trip，且本地选择不加载 snapshot。
- bindgen CLI 单元测试 — 4 passed；Kotlin/Swift bindings 已从最终 release library
  原地重生成，generated contract 包含 callback owner 和清零断言。
- `Gradle 8.7 --no-daemon :app:testDebugUnitTest` — 39 passed；覆盖 desired/draft
  状态、并发 action gate、注册/撤销调和、credential 清理、当前 URI 权威与恢复。
- `Gradle 8.7 --no-daemon :app:connectedDebugAndroidTest` — 30 passed / 0 skipped /
  0 failed；覆盖真实 API 34 Keystore、失效分类、unlock state transitions、
  Compose 状态、clearable editor、真实 `ContentResolver` 及云式 pipe key-file 交接。
- `x86_64-linux-android` 与 `aarch64-linux-android` release native library 均成功构建，
  新增符号均存在；debug/release APK 成功组装。
- `lintDebug` 与 `lintVitalRelease` 成功，0 errors / 5 warnings。warning 为
  `allowBackup` 的新规则提示、生成绑定的 3 个 API-floor 冗余检查和待 UI 收口的 app
  icon，不是 M1 功能或密钥材料问题。

## 自审

自审摘要：PR 的 6 条评论全部按所有权、状态定义、权威源和 fresh-credential
不变量整体修复；评论修复后的第二遍 002/003/D5/007 终审又发现并修复 6 项：
SAF key-file 的 `/proc` 规范化失效、无关 URI 阻塞当前 vault、设置失败丢草稿、
biometric 分支遗留未提交密码、sensitive factory 复制异常未清源缓冲、存在但已偏离
URI 权威的工作副本未重建。最终 diff 复核未发现仍可由本轮自行定位的未修问题。
