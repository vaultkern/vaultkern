## 已交付

- 以 Android Keystore AES-256-GCM 保护 OneDrive refresh token，StrongBox 可用时优先、
  否则回退；密文、安全级别和随机 key alias 存于 app-private `AtomicFile`。写入采用新 key
  → 原子密文提交 → 清旧 key，启动/每次调和清理 `.new` 与孤儿 key；坏 tag、坏记录、
  缺 key 会删除密文和 key 并要求重新连接账户。
- token 在 adapter 中只以可清零 UTF-8 byte buffer 进出；补了两个独立的小型共享绑定
  提交：byte-buffer 构造/复制，以及 UniFFI callback lower 后强制关闭 sensitive owner。
  生成器会稳定重现后者，unlock blob 返回缓冲也同时修复；没有改变现有接口形状。
- Compose 解锁页支持发起/完成 loopback PKCE 登录、浏览文件夹、选择 KDBX source 与
  `unlockCurrent`；账户、source、sync、entry edit/save 全部直接使用 resident app 的同一
  `VaultSession` 和 writer lock，没有 outbox、overlay 进程或 Android 侧同步语义。
- 已解锁的 OneDrive vault 显示 `online` / `cache` / `pending_sync`、last error、显式重试
  和 recoverable conflict-copy 提示。进程重建会从核心状态恢复当前远端 source、pending
  状态与 sync 按钮；登录 token 已提交但 UI 尚未更新时，也会识别为已连接。
- 保存与 retry 复用核心现有 ETag CAS、synced base、三方 field patch、ambiguous PUT
  readback、terminal conflict-copy 去重和本地 pre-rename fingerprint 路径。

## 停放项 1：SIGKILL 发生在首次远端 PUT 窗口

现有核心对网络返回 `OutcomeUnknown` 后能 readback，并在 readback 不可用时写 durable
pending candidate；测试覆盖这一路径。但 `save_onedrive_vault` 当前先调用
`conditional_write`，之后才在 outcome/error 分支保存 pending candidate 与 observed base。
如果 Android 进程在 PUT 已发出、该分支执行前被杀：服务器若提交，重启可拉到新 head；
服务器若未提交，本地编辑只在已死亡进程内存中，无法恢复。这不满足“任意点被杀”。

修复需要核心在 PUT 前 durable-stage candidate、observed base 与 operation receipt，并让
启动恢复判定远端是否提交；这是新的核心同步事务语义，不是 Android adapter 能补的，
且任务明确要求 M4 复用现有核心、不新增语义，所以本 PR 不越界修改。

## 停放项 2：真实账户 ceremony / 取消

本地构建环境未设置 `VAULTKERN_ONEDRIVE_CLIENT_ID`，也没有可用 Microsoft 测试账户，
因此当前 APK 的真实登录、Graph ETag PUT 和物理网络中断没有设备证据。另一个既有核心
限制是 loopback listener 在 `accept()` 上无取消/超时；用户放弃浏览器授权后，只能等待
并最终重启进程释放端口。要达到可取消的产品 ceremony，需要核心提供可取消/可轮询的
pending login，而不能用 Android UI 假装取消成功。
