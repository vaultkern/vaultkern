- 日期：2026-07-23（Asia/Shanghai）
- 分支：`codex/android-m4-sync`，堆叠在 M3 / PR #47
- 契约：`docs/design/000` r16、`002`、`003` r15、`007`
- 结论：Android 文件选择、私有工作副本、token/account/source/sync adapter、进程重建恢复
  和 fallback UX 已实现。真实 Microsoft ceremony 尚无环境证据；此外，现有核心仍不能保证

## 已交付

- 用户选择本地 `content://` 文档后，该 URI 始终是权威文件。Android 持久化读写权限，
  只在 app-private 目录维护供核心执行原子事务的工作副本；普通保存、passkey 注册保存和
  崩溃恢复都发布回同一个 URI。发布前重新读取权威文件并比较 fingerprint；外国修改转成
  sibling conflict copy，无法创建 sibling 时保留可恢复的私有 conflict copy。
- 用户选择 OneDrive KDBX 后，先把 source 设为核心 current reference，再通过新增的
  `VaultSources.preloadCurrentVault()` 小型 additive UniFFI 入口立即下载到核心的
  app-private `resident-state/remote-cache`。下载失败仍保留已选择的 current reference，
  明确提示重试；重启后可从 current reference 继续预载/解锁，不会退回旧的本地文件。
- 进程重建从核心唯一的 current reference 恢复本地或远程选择，不依赖 Compose 内存状态；
  current reference 缺失时会一次清空所有陈旧的本地路径、远程标记和 sync 投影。切换 vault
  同时刷新该 vault 的 quick-unlock enrollment 与 Keystore security level，避免沿用前一个
  vault 的实际状态。
- OneDrive refresh token 由 Android Keystore AES-256-GCM 保护，StrongBox 可用时优先、
  否则回退；密文、安全级别和随机 key alias 存于 app-private `AtomicFile`。写入采用新 key
  → 原子密文提交 → 清旧 key，启动/每次调和清理 `.new` 与孤儿 key；坏 tag、坏记录、
  缺 key 会删除密文和 key 并要求重新连接账户。token 提交、读取、删除和存储调和在同一
  adapter 锁内串行，调和不会把尚未提交记录的新 key 当作孤儿删除。
- token 在 adapter 中只以可清零 UTF-8 byte buffer 进出；共享 binding generator 在
  callback lower 后强制关闭 sensitive owner，unlock blob 返回缓冲也使用同一路径。
  所有 UniFFI 扩展均为小型 additive commit，没有重塑现有模块接口。
- Compose 目前只做必要接线：发起/完成 loopback PKCE 登录、浏览文件夹、选择 KDBX、
  `unlockCurrent`、显式 sync/retry 和 conflict-copy 提示。账户、Autofill、Credential
  Manager、source、sync 与编辑保存全部直接使用 resident app 的同一 `VaultSession` 和
  writer lock；没有 Android outbox、overlay 进程或第二套同步语义。
- 保存与 retry 复用核心现有 ETag CAS、synced base、三方 field patch、ambiguous PUT
  readback、terminal conflict-copy 去重和本地 pre-rename fingerprint 路径。

## 停放项 1：SIGKILL 发生在首次远端 PUT 窗口

现有核心对网络返回 `OutcomeUnknown` 后能 readback，并在 readback 不可用时写 durable
pending candidate；测试覆盖这一路径。但 `save_onedrive_vault` 当前先调用
`conditional_write`，之后才在 outcome/error 分支保存 pending candidate 与 observed base。
如果 Android 进程在 PUT 已发出、该分支执行前被杀：服务器若提交，重启可拉到新 head；
服务器若未提交，本地编辑只在已死亡进程内存中，无法恢复。这不满足“任意点被杀”。

修复需要核心在 PUT 前 durable-stage candidate、observed base 与 operation receipt，并让
启动恢复判定远端是否提交；这是新的核心同步事务语义，不是 Android adapter 能补的，
且任务明确要求 M4 复用现有核心、不新增语义，所以本 PR 不越界修改。

## 停放项 2：真实账户 ceremony / 取消

本地构建环境未设置 `VAULTKERN_ONEDRIVE_CLIENT_ID`，也没有可用 Microsoft 测试账户，
因此当前 APK 的真实登录、Graph ETag PUT 和物理网络中断没有设备证据。代码逻辑与模拟
核心契约已验证，但不把它替代成真实服务证据。另一个既有核心限制是 loopback listener
在 `accept()` 上无取消/超时；用户放弃浏览器授权后，只能等待并最终重启进程释放端口。
达到可取消的产品 ceremony 需要核心提供可取消/可轮询的 pending login，Android UI 不能
伪造取消成功。
