# Android M2 — resident app

- 日期：2026-07-22（Asia/Shanghai）
- 分支：`codex/android-m2-resident-app`，堆叠在 M1 / PR #43
- 契约：`docs/design/000` r16、`002`、`003` r15、`007`
- 结论：resident browse/edit/durable-save 与 conflict-copy 主链完成；生产级 Android
  文件选择/所有权仍需一条新契约规则，按停止条件停放，详见“停放项”。

## 交付

- Compose shell 在交互或 quick unlock 成功后从同一个进程内 `VaultSession` 列出、
  读取并编辑 entry；保存直接调用既有 `editEntry` + `save` UniFFI 模块接口。
- adapter 会完整搬运 title/username/password/URL/notes/TOTP URI/custom fields，所有
  UniFFI sensitive owners 在成功、错误路径都关闭；entry/password/TOTP/custom field
  的诊断输出全部遮蔽，成功/取消编辑后 UI 不再持有 draft。
- 保存结果显式呈现 `SAVED`、`SAVED_TO_CACHE`、field-patch 与 `CONFLICT_COPY`；
  conflict-copy 路径保留在 UI，绝不把冲突伪装成普通成功。
- 复用既有 writer lock、temp write/readback/fsync、发布前目标指纹复核、atomic rename、
  target/目录 fsync 与 sibling conflict-copy 路径，没有在 Android 层另造保存语义。
- Android SELinux 禁止 Unix hard-link 备份，因此 runtime 的 Android-only adapter 改为
  复用现有 Windows durable-copy 备份策略：copy + metadata + file fsync + directory
  fsync；Linux/macOS hard-link 路径与 Windows 分支行为保持不变。

## 停放项：Android 用户文档所有权

Android 14 的 Storage Access Framework 给应用的是可持久化 `content://`/文件描述符
能力，不是可安全解析的真实路径；现有 local-source API 的安全性来自对目标目录内
temp/backup/rename/fsync 的控制。把 SAF URI 偷换成 `/proc/self/fd` 或猜测
`/storage/emulated/0` 会丢失权限、atomic rename 与 provider 一致性保证。

继续需要契约明确选择其一：

1. 选择文档后复制为 VaultKern 管理的 app-private vault（“打开即迁移”），并定义
   escape-hatch/卸载行为；或
2. 新增 SAF document-provider source，定义 provider 不支持原子替换/目录 fsync 时的
   失败与 conflict-copy 规则。

本 PR 没有擅自选择。当前 Compose 路径入口与设备测试针对 app-private/已配置 local
source；因此不声称普通用户已能从系统 Files 选择任意 KDBX 并原位安全保存。
