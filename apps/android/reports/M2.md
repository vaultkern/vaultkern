# Android M2 — resident app

- 日期：2026-07-23（Asia/Shanghai）
- 分支：`codex/android-m2-resident-app`，堆叠在 M1 / PR #43
- 契约：`docs/design/000` r16、`002`、`003` r15、`007`
- 结论：resident browse/edit/save、系统文件选择与同一 URI 写回链路完成；任意 SAF
  provider 不提供原子 rename/目录 fsync 保证，因此“用户 URI 上的 007 原子替换”作为
  OS 限制停放，但不会阻塞后续里程碑。

## 交付

- Compose shell 使用系统 `ACTION_OPEN_DOCUMENT` 选择 KDBX，并持久保留该文档的
  read/write URI capability；用户选择的 URI 始终是权威文件，不把它迁移成
  app-private vault。
- 选择时逐字节建立 app-private 事务工作副本。它只供 resident core、进程内
  Autofill/Credential Provider 与崩溃恢复共用；保存完成后写回用户选择的同一 URI。
- resident `VaultSession` 继续走既有 `editEntry` + `save` UniFFI 路径，复用 core 的
  writer lock、写临时文件、fsync、发布前内容指纹检查、atomic rename、目录 fsync 与
  conflict-copy；Android adapter 没有引入新的 KDBX 或 merge 语义。
- URI 发布有持久 `pending` receipt 与 `writeStarted` 阶段，并在每一步读回核对内容：
  进程可死在 core save、provider 写前、写中、写后或 binding 提交后，下一次启动都能
  区分未写、已写但返回不明、读回截断和已提交。进程内 core save 还持有 active guard，
  后台调和不能在 core 原子发布前把 receipt 当作 no-op 提前清掉。
- 外部内容在发布前变化时不覆盖原 URI：优先请 provider 创建 sibling conflict copy；
  provider 不授予父目录创建能力时保留 app-private recoverable conflict，并把工作副本
  脱离旧 URI。仅 `last_modified` 变化而内容相同不会误分叉。
- provider 暂时失败或读回不匹配会返回 `SAVED_TO_CACHE` 并保留 receipt，绝不把只存在
  于 private 工作副本的成功伪装成已发布；重新选择同一 URI 也不能覆盖 pending 候选。
  ambiguous write 后若权威内容既非 baseline 也非候选，一律走 conflict-copy，不用重试
  覆盖来猜测它是否只是本次写入的截断结果。
- quick unlock 调 core blob 前先刷新 URI 权威内容到 private 工作副本，确保外部换盐会
  触发文件 HMAC miss 与 app 内 KDF refresh，而不是命中陈旧副本。
- entry 的 browse/read/edit/save 全部在同一 app 进程与同一 `VaultSession` 内完成；
  sensitive UniFFI owners 在成功/异常路径关闭，KDBX 临时 byte buffers 在读取、哈希、
  写回及元数据查询失败路径清零，诊断输出遮蔽密码、vault 路径和显示名。
- Android SELinux 禁止 Unix hard-link 备份；此前 M2 core 适配继续使用已验证的
  Android durable-copy 备份策略（copy + metadata + file fsync + directory fsync），
  不改变 Linux/macOS/Windows 分支。
- app-private receipt、binding 与工作副本只接受 `ATOMIC_MOVE`，目录 fsync 失败直接向上
  报错，不再静默退化；private conflict-copy 以候选内容指纹形成稳定唯一文件名，重放
  同一候选幂等，不同候选不会覆盖更早的可恢复副本。

## 进程死亡恢复

1. `prepareSave` 原子落盘 private 工作副本的保存前指纹。
2. core 对 private path 执行 007 durable save；若进程在此处死亡，core 先恢复自身
   pre-rename 状态，URI receipt 再判断工作副本是否实际变化。
3. 写回前重新读取 URI 并与选择时 baseline 比较；不一致走 conflict-copy。
4. 写 provider 前持久提交 `writeStarted`，写后重新打开同一 URI 核对候选内容。
5. 启动调和识别原内容、候选内容、失败/截断内容与外部内容；只有权威内容仍等于
   baseline 时才重放写入，其他歧义落入 conflict-copy，绝不静默覆盖。

## OS 限制停放项

Android SAF 只保证持久 URI capability 和 provider 文件描述符；`rwt` 的实际 truncate/
write 实现由 provider 决定，单文档 grant 也不保证能在父目录创建 temp 并 atomic rename。
因此，任意第三方 provider 上无法证明与 core 本地路径完全相同的“temp + fsync + rename +
directory fsync”原子性。

当前实现已把可做到的安全边界做实：private core commit 是原子且 durable；URI 写回有
持久 intent、写前内容指纹、写后 readback、幂等恢复与 conflict-copy fallback。它不会
静默丢数据或伪报发布成功，但本 PR 不把这些补偿机制声称为 provider 原子 rename。
