# Android M3 — credentials

- 日期：2026-07-22（Asia/Shanghai）
- 分支：`codex/android-m3-credentials`，堆叠在 M2 / PR #46
- 契约：`docs/design/000` r16、`002`、`003` r15、`007`
- 结论：M3 验收项通过；未修改 Rust、UniFFI 接口或其他平台代码。

## 交付

- `AutofillService` 与 Credential Manager `CredentialProviderService` 均未声明
  `android:process`，由 resident `VaultKernApplication` 直接复用同一个 `VaultSession`、
  writer lock、unlock adapter 与正常保存路径；没有 outbox 或 overlay 进程。
- Autofill 在 dataset 级别要求认证，fresh UV 后从核心直接读取 password 与当前 TOTP；
  多候选时在 `FLAG_SECURE` Compose picker 中选择。取消、读取失败或不支持保存捕获时
  显式失败，不会返回空成功或宣称已保存。
- Passkey 注册严格解析 WebAuthn creation options、执行 RP/origin 校验与 biometric UV，
  通过既有核心构造写入 KPEX-format entry，并在生成响应前走正常 durable save；断言从
  核心列出候选、验证 allow-list/RP、由存储的私钥签名并返回核心 authenticator data。
- native 调用以当前签名证书计算 Android origin，并通过 Google Digital Asset Links API
  验证 `delegate_permission/common.handle_all_urls`；浏览器调用必须匹配受信 allowlist，
  使用调用方提供的 32-byte `clientDataHash` 和规范允许的 placeholder `clientDataJSON`。
  allowlist 固定为 2026-07-22 Google 公共快照，SHA-256
  `594a83e3cb3475e8af8da22595b3481cbefb61b747c7a67a66aa2e47433ed79c`。
- 每个系统 credential entry 都绑定 request JSON 与 `clientDataHash` 的摘要，多个 option /
  challenge 并存时不会误签另一个请求。认证后选择超过 30 秒会重新要求 UV；失败或取消
  不刷新 freshness。注册、断言所有异常路径都会结束核心 operation，未提交注册回滚。
- 系统 provider 的启用仍由 Android 设置保存为用户期望状态；应用内没有“开关即执行”
  的旁路副作用。Kotlin 不接触 passkey 私钥；响应和诊断对象遮蔽 credential/TOTP/password，
  UniFFI sensitive owners 在 reveal 后立即关闭，明文只保留到 OS dataset 交接边界。

  Autofill API 34 dataset population、真实核心 password/TOTP population、两项 service
  与 resident app 同进程声明、Google allowlist checksum，以及“注册 durable save → 关闭
  session → 重开 KDBX → 列出 KPEX passkey → assertion signature + UV flag”往返。
- `Gradle 8.7 --no-daemon :app:assembleDebug :app:assembleRelease` — 均成功；release
  `lintVital` 通过，APK 保留 M1/M2 的 x86_64 与 arm64-v8a UniFFI 核心。

M0 已证明 Android 14 系统 Credential Manager provider 的 create/get ceremony 和
Autofill cold-start/dataset-auth 路径；按任务要求直接沿用、未重跑。M3 产品测试证明真实
KDBX/KPEX 核心往返，但此 AVD 没有可控的外部 HTTPS RP、生产签名与 Chrome 登录态，
因此不把它表述为生产网站 Digital Asset Links 端到端证明。

## 进程退出边界

- 注册在 commit 前退出：磁盘不含新 passkey；commit 后、系统收到 response 前退出：
  已 durable-save 的 KPEX entry 保留，调用方看到取消/未知并可重新发起 ceremony。
- 断言只读；任意退出都不创建旁路队列。核心 operation 在正常取消/错误时显式结束，
  进程被系统终止时随 resident process 一并释放。
